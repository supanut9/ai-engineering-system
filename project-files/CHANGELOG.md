# CHANGELOG.md

## Purpose

Tracks meaningful project changes over time.

## Should Exist

Yes, for every real project.

## Typical Contents

Use `Keep a Changelog` style.

Recommended structure:

- `Unreleased`
- released version sections with dates
- categorized entries under:
  - `Added`
  - `Changed`
  - `Deprecated`
  - `Removed`
  - `Fixed`
  - `Security`

## Suggested Outline

```md
# Changelog

All notable changes to this project will be documented in this file.

The format is based on Keep a Changelog.

## [Unreleased]

### Added

- ...

### Changed

- ...

### Fixed

- ...

## [1.0.0] - 2026-05-08

### Added

- Initial release.
```

## Rules

- keep `Unreleased` at the top
- record user-visible or maintainer-important changes, not every commit
- prefer concise bullets
- group related changes together

## Naming

Use `CHANGELOG.md` in uppercase.
