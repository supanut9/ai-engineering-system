# data model — hello-todo-fastify

This document describes the logical data model for `hello-todo-fastify`. The in-memory
store (`Map<string, Todo>` inside the memory adapter) is an implementation detail
and is not part of the model.

---

## entity: Todo

| field | type | required | default | notes |
|---|---|---|---|---|
| `id` | string | yes (server-assigned) | — | UUID v4 generated by the `uuid` package at creation time; callers must treat as opaque |
| `title` | string | yes | — | non-empty after trim; max 200 characters; trimmed before storage |
| `completed` | boolean | yes | `false` | indicates whether the todo is done |
| `due_at` | string \| null | no | `null` | RFC3339 UTC timestamp; null when no due date is set |
| `created_at` | string | yes (server-managed) | current UTC time at creation | RFC3339; immutable after creation |
| `updated_at` | string | yes (server-managed) | same as `created_at` at creation | RFC3339; updated on every successful PATCH |

---

## validation rules

| field | rule |
|---|---|
| `title` | must be present; must be non-empty after trimming leading/trailing whitespace; trimmed value must be ≤ 200 characters |
| `due_at` | if present and non-null, must be a valid RFC3339 timestamp; empty string is not valid |
| `id` | immutable; supplied by server only; client-supplied `id` on create is ignored |
| `created_at` | immutable; set by server at creation; client-supplied value is ignored |
| `updated_at` | set by server on creation and on every successful update; client-supplied value is ignored |

---

## notes

- there is no separate persistence schema; the same `Todo` interface is used in-memory
  and serialized to JSON in API responses
- the TypeScript interface lives at `src/core/todo/todo.ts`
- timestamps are stored and transmitted as ISO 8601 / RFC3339 strings (e.g.
  `"2026-05-16T10:00:00.000Z"`). Using strings rather than `Date` objects avoids
  serialization edge cases in JSON.stringify and keeps the wire format explicit.
- `null` is the JSON representation of an absent `due_at`; the TypeScript type uses
  `string | null` to distinguish "not set" from an empty string (which is invalid)
- id generation uses UUID v4 (`uuid` package) rather than a custom random generator.
  UUIDs are not time-ordered but are globally unique and widely understood; this is
  sufficient for an in-memory example. A ULID or UUIDv7 can be substituted in v0.2.0
  if insertion-order ids are needed.
