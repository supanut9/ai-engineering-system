# data model — hello-todo-nextjs

This document describes the logical data model for `hello-todo-nextjs`. The in-memory
store (`Map<string, Todo>` inside `src/lib/repo.ts`) is an implementation detail
and is not part of the model.

---

## entity: Todo

| field | type | required | default | notes |
|---|---|---|---|---|
| `id` | string | yes (server-assigned) | — | 32-char hex; generated by `crypto.randomBytes(16).toString('hex')`; callers must treat as opaque |
| `title` | string | yes | — | non-empty after trim; max 200 characters; trimmed before storage |
| `completed` | boolean | yes | `false` | indicates whether the todo is done |
| `due_at` | string \| null | no | `null` | ISO8601 UTC timestamp; null when no due date is set |
| `created_at` | string | yes (server-managed) | current UTC time at creation as ISO8601 | immutable after creation |
| `updated_at` | string | yes (server-managed) | same as `created_at` at creation | updated on every successful PATCH |

---

## validation rules

| field | rule |
|---|---|
| `title` | must be present; must be non-empty after trimming leading/trailing whitespace; trimmed value must be ≤ 200 characters |
| `due_at` | if present and non-null, must be a valid ISO8601 timestamp; empty string is not valid |
| `id` | immutable; supplied by server only; client-supplied `id` on create is ignored |
| `created_at` | immutable; set by server at creation; client-supplied value is ignored |
| `updated_at` | set by server on creation and on every successful update; client-supplied value is ignored |

---

## notes

- there is no separate persistence schema; the same `Todo` interface is used in-memory
  and serialized to JSON in API responses
- the `Todo` type lives at `src/types/todo.ts`
- id generation lives in `repo.generateId()` using Node's `crypto` stdlib
- `null` is the JSON representation of an absent `due_at`; the TypeScript type uses
  `string | null` to distinguish "not set" from an empty value
- all timestamps are ISO8601 strings (JavaScript `new Date().toISOString()`) rather
  than `Date` objects to avoid serialization inconsistency
