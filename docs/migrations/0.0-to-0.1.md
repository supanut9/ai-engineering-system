# 0.0 to 0.1 migration

> **Status: scaffold.** This page is a scaffold pending the v0.1.0 release. Content will be finalized when the release ships and any last-minute gaps are known.

## summary

`0.1.0` is the first tagged release of the AI Engineering System. There is no version `0.0.x` to upgrade from in the traditional sense — the `0.0` baseline is the state of the repo before any release was tagged. This migration guide is therefore an adoption guide: it walks a project from "no system" to "AES-bootstrapped" for the first time.

If your project already uses the system from an untagged checkout, this page also lists the known gaps in `0.1.0` and what, if anything, requires a manual fix.

## pre-flight

Before running the bootstrap:

- Read the [docs landing page](../index.md) for the top-level overview and Quickstart.
- Read [`docs/quickstart.md`](../quickstart.md) for the step-by-step first-run guide.
- Decide which stack fits your project. Available in `0.1.0`:
  - `go-gin-layered`
  - `go-gin-clean`
  - `go-gin-hexagonal`
  - `nestjs-layered`
  - `nextjs-default`
- Decide which agent adapter you want: `claude`, `codex`, or `both`.

## steps

1. Obtain the system. Either clone the repo directly or download the `v0.1.0` release archive (URL published in the GitHub release when `0.1.0` ships):

   ```bash
   git clone https://github.com/<owner>/ai-engineering-system.git
   cd ai-engineering-system
   ```

2. Run the bootstrap script:

   ```bash
   ./scripts/init-project.sh --name <project> --stack <stack> --agent <claude|codex|both>
   ```

   The script creates a new directory at `./<project>/` with workflow artifacts, adapter files, tooling configs, and a skeleton source tree appropriate to the chosen stack.

3. Open the bootstrapped `.ai/workflow/project-context.md` and complete the Phase 0 intake fields. Alternatively, open the project in Claude Code and run the `project-intake` skill, which walks the intake interactively.

4. Adopt tooling configs from `tooling/<stack>/` if you want enforceable lint and format rules. CI templates for GitHub Actions ship at `tooling/ci/github/` and can be copied into `.github/workflows/` in the bootstrapped project.

5. Run the doctor script from inside the bootstrapped project to confirm required files are in place:

   ```bash
   ./scripts/doctor.sh --target ./<project>
   ```

6. Commit and push the bootstrapped project. Future updates to adapter files (CLAUDE.md, .codex/, etc.) can be pulled with:

   ```bash
   ./scripts/sync-agent-files.sh --source <path-to-system> --target <path-to-project>
   ```

## known gaps in 0.1.0

These are issues present at the `0.1.0` release that require manual attention. They will be addressed in subsequent releases.

- **GitHub owner placeholders:** when forking, replace `supanut9` with your own GitHub handle in `mkdocs.yml` (`site_url`, `repo_url`), `docs/quickstart.md`, `docs/index.md`, and `.github/ISSUE_TEMPLATE/config.yml`. The upstream `0.1.0` ships configured for `supanut9/ai-engineering-system`.

- **`mkdocs.yml` site placeholders:** `site_url` and `repo_url` in `mkdocs.yml` are placeholder values (`https://example.com`). Update them to your actual GH Pages URL and repo URL before deploying the docs site.

- **`scripts/doctor.sh` is advisory only:** the doctor script reports drift but does not auto-fix anything. Review its output and apply corrections manually.

- **React Native + Expo selftest is slow:** the selftest matrix runs the React Native skeleton on-demand only (not on every push) because the install is heavy. If you are working with that stack, run `./scripts/selftest.sh --stacks react-native-expo` explicitly before opening a PR.
