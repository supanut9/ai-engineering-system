# Multi-Service Api And Web

## Use When

Use this blueprint when:

- the project ships **two or more deployable services** that are versioned and
  released together (typically an HTTP API plus a web front-end);
- the services share a single repository, a single CI pipeline, and a single
  ADR / requirements stream;
- shared concerns (env-var conventions, lint config, release process) outweigh
  the cost of cross-service coordination.

**Do not** use this blueprint when:

- the services have independent release cadences, owners, or branching models тАФ
  use separate repos bootstrapped from the single-service blueprints;
- the project has only one service today; add the multi-service layout when the
  second service actually arrives, not speculatively;
- the second service is a sidecar, worker, or batch job that runs inside the
  same deploy unit as the API тАФ that belongs inside the API's repo, not a
  sibling app.

## Stack

Multi-service is an **organisational pattern**, not a stack. Each `apps/<name>`
directory follows one of the existing single-service blueprints:

- `apps/api/` тЖТ any of `go-gin-{layered,clean,hexagonal}`, `nestjs-layered`,
  `fastify-hexagonal`, `fastapi-layered`.
- `apps/web/` тЖТ `nextjs-default`, or `react-native-expo` (named `apps/mobile/`).
- additional services as `apps/<name>/` тАФ only when they meet the "ship together"
  rule above.

## Code Architecture

- top-level: monorepo with `apps/` (deployables), optional `packages/` (shared
  libraries), and root-level orchestration (Makefile, CI, docs);
- each `apps/<name>/` follows its own blueprint's architecture independently;
- shared code lives under `packages/<name>/` only when at least two apps consume
  it. Premature `packages/` directories are a smell тАФ copy-paste twice before
  extracting.

## Bootstrap

This blueprint is composed, not generated by `init-project.sh` directly. Run the
single-service bootstrap once per app, then add the root-level files described
below.

```bash
# 1. Create the umbrella directory and move into it.
mkdir my-product && cd my-product

# 2. Bootstrap the API into apps/api/.
$SYSTEM/scripts/init-project.sh \
  --name api --stack go-gin-hexagonal --agent claude \
  --target apps/api --no-git

# 3. Bootstrap the web app into apps/web/.
$SYSTEM/scripts/init-project.sh \
  --name web --stack nextjs-default --agent claude \
  --target apps/web --no-git

# 4. Add the root files (see "Root Files" below).

# 5. git init once at the umbrella root.
git init
git add . && git commit -m "chore: bootstrap multi-service from ai-engineering-system"
```

The `--no-git` flag on each per-app run is required тАФ git is initialised once at
the umbrella root, not inside each `apps/<name>/`.

See [`bootstrap-multi-service.md`](../../docs/cookbook/bootstrap-multi-service.md)
for the full step-by-step recipe with the root file contents.

## Folder Structure

```text
my-product/
  apps/
    api/                  # bootstrapped from a single-service stack
      cmd/ internal/ go.mod ...
      .ai/workflow/       # per-app workflow state
      CLAUDE.md           # per-app adapter
    web/                  # bootstrapped from a single-service stack
      src/ package.json ...
      .ai/workflow/
      CLAUDE.md
  packages/               # optional тАФ only when 2+ apps share code
  docs/                   # repo-wide docs (cross-app architecture, ADRs)
    requirements/
    architecture/
      decisions/
    plan/
    tests/
    release/
    maintenance/
  .ai/                    # umbrella workflow state (cross-app)
    workflow/
      project-context.md
      workflow-state.md
      service-map.md
  Makefile                # delegates to apps/*/Makefile
  CLAUDE.md               # umbrella adapter тАФ points at per-app adapters
  .github/workflows/      # one CI workflow that runs each app's checks
  README.md
  CHANGELOG.md
  .gitignore
  .env.example
```

## Folder Responsibilities

- `apps/<name>/` = one independently-buildable service, owns its own
  dependencies, lint config, and tests.
- `packages/<name>/` = shared code consumed by тЙе2 apps; only created on demand.
- `docs/` at the umbrella root = decisions that span apps (architecture, release
  plan, cross-app test strategy). Per-app concerns stay inside the app's `docs/`.
- `.ai/workflow/` at the umbrella root = product-level workflow state. Each app
  also has its own `.ai/workflow/` for service-level state.
- `Makefile` at the umbrella root = thin orchestrator. Per-app targets (`test`,
  `lint`, `build`, `run`) recurse into each app's Makefile via
  `$(MAKE) -C apps/<name> <target>`.
- `.github/workflows/` at the umbrella root = single CI workflow per concern
  (e.g. `ci.yml`), each job scoping to one app's `working-directory`. Avoid
  duplicating per-app CI files at the umbrella root.

## Required Workflow Files

At the umbrella root:

- `CLAUDE.md` (or `.codex/`) тАФ points at per-app adapters and lists each app.
- `.ai/workflow/project-context.md` тАФ product goal, app inventory, owner.
- `.ai/workflow/workflow-state.md` тАФ product-level phase tracking.
- `.ai/workflow/service-map.md` тАФ explicit list of every `apps/<name>/`, its
  blueprint, port, and which other apps depend on it.

Per app (unchanged from the single-service blueprint):

- `apps/<name>/CLAUDE.md`, `apps/<name>/.ai/workflow/*`.

## Root Files

The umbrella root needs five files that `init-project.sh` does not generate when
invoked twice. See
[`templates/docs/multi-service-root-readme.md`](../../templates/docs/multi-service-root-readme.md)
and the cookbook recipe for filled-in versions:

1. `README.md` тАФ top-level product README that lists each app and points at
   per-app READMEs.
2. `Makefile` тАФ orchestrates per-app targets.
3. `.gitignore` тАФ union of each app's ignores. Each `apps/<name>/.gitignore` is
   redundant once the root `.gitignore` covers its patterns; keep the root file
   authoritative.
4. `.env.example` тАФ documents env vars that span apps (e.g. `API_URL` consumed
   by the web app). Per-app `.env.example` files keep their own service-only
   vars.
5. `CLAUDE.md` (or `.codex/`) at the umbrella root тАФ instructs the agent to
   read both per-app adapters before doing cross-app work, and to update the
   product-level workflow state alongside per-app state.

## Notes

- The umbrella does not need its own stack tooling. Lint, format, and test
  configs live inside each `apps/<name>/`. The umbrella's only tooling concern
  is the root `Makefile` and the shared CI workflow.
- Versioning is product-level by default: one `CHANGELOG.md` at the root, one
  `VERSION` / release-please manifest at the root. Per-app version files belong
  only when the apps genuinely ship on independent cadences тАФ at which point
  the multi-service blueprint is the wrong choice and they should be split into
  separate repos.
- The `service-map.md` is the single source of truth for "which app talks to
  which". Update it whenever an app is added or a dependency changes; the
  `workflow-runner` skill reads it when it needs to reason about cross-app
  effects.
- Resist adding a third app. Most monorepo regrets start with the third app
  being added because it was easier than spinning up a new repo, not because
  the "ship together" rule applied.
